The 3-Step Workflow

How the AI builds your screen behind the scenes.

Step 1: plan_feature()

Triggered automatically when you say "create", "build", "design", or mention a feature name.

// You type: "Create Event Overview Dashboard for Log360 Cloud" // AI calls internally: plan_feature( feature_name = "Event Overview Dashboard", description = "Dashboard showing event metrics, trends..." ) // Returns: feature_plan_id + shell_guide table

Step 2: get_atoms()

AI picks the best shell and fetches all web component templates + layout rules.

get_atoms( feature_plan_id = "abc123", shell = "A", // Dashboard shell filename = "event-overview-dashboard.html" ) // Returns: 31 CDN atoms + shell_contract + scaffold + CSS tokens

Step 3: Write HTML

AI assembles the page using <me-*> web component tags and writes the file.

Zero Code from You

You don't write any code! Just describe what you want, and the AI handles the entire pipeline.

Header

A header is the topmost section of an application screen. It typically contains essential information and navigation elements.

Overview

The Header (<me-page-header>) serves as a vital component of the application interface, providing users with a clear and consistent navigation framework. It enhances user experience by displaying essential information and offering quick access to key features, ensuring that users can easily find what they need while browsing.

Anatomy

The header is composed of key parts that work together to provide navigation and context.

Actionbar

The Actionbar sits directly above a data table, combining search, pagination, and optional action buttons in one compact 40px bar.

Overview

The <me-actionbar> is a conjoined twin of <me-data-table>. CSS auto-collapses the gap between them. It handles search filtering, pagination state, and exposes a slot for custom action buttons on the left side.

Live Preview
<me-actionbar total="1250" page-size="25"> <me-ab-btn icon="plus" label="Add Rule"></me-ab-btn> <me-ab-btn icon="download" label="Export"></me-ab-btn> </me-actionbar>

Anatomy

The actionbar is composed of left-side controls (search + action buttons) and right-side controls (count + pagination).

1 2 3 4
1. SearchFull-text filter input — filters visible rows in real time.
2. Action ButtonsOptional <me-ab-btn> children — rendered left of search. Use for Add, Export, Import actions.
3. Record CountShows "1 — 25 of 1,250" driven by total and page-size attrs.
4. PaginationPrevious / Next controls. Page count derived from total ÷ page-size.

Data Table

The Data Table renders sortable, selectable tabular data. It is always paired with <me-actionbar> as a direct sibling above it.

Overview

The <me-data-table> web component produces a full-featured table with optional checkboxes, sortable columns, and responsive row hover states. Column headers are defined as <me-col> children and row data is placed inside a <template> tag (required by the HTML parser to prevent bare <tr> stripping).

Elevation

Elevation communicates the relative depth of a component using shadows. Higher elevation means closer to the user — drawing attention and establishing visual hierarchy across the interface.

Overview

The Elegant Design System uses a six-level elevation scale. Each level is defined by a precise box-shadow value using a single unified shadow colour #000000 · 16%. Elevation reinforces component layering, interaction state, and spatial relationship across the product.

Z-axis depth
Higher shadow = visually closer to the viewer on the Z-axis
Unified colour
All shadows use #000 at 16% opacity for consistent depth perception
Contextual
Match shadow to background — each level targets a specific surface tone
Interactive
Elevation shifts on hover/focus to communicate interactivity and state

Usage Guidelines

Do
  • Use higher elevation for components that float above the current content layer
  • Pair shadow level with the correct z-index to maintain visual consistency
  • Apply Card Hover shadow on mouse-enter for interactive cards
  • Use Drawer shadow (lateral) when overlaying content from the right
  • Match shadow intensity to the background — lighter backgrounds need lighter shadows
Don't
  • Don't stack the same elevation level on overlapping components
  • Avoid using custom shadow values — always use the defined tokens
  • Don't apply Level 5 (Drawer) shadow to non-drawer components
  • Never use coloured shadows — the system only uses #000 · 16%
  • Don't add elevation to flat surfaces like table rows or page backgrounds
